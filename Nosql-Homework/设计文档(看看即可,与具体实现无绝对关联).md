# GitHub 开源项目分析平台设计文档

## 1. 项目概述

### 1.1 项目名称

GitHub 开源项目分析平台

### 1.2 项目背景

随着开源软件快速发展，GitHub 已成为全球最大的代码托管平台之一，沉淀了大量项目、社区协作与技术演进数据。高校科研与软件工程研究需要可持续、可验证、可复现的数据平台来支撑研究结论。

本项目目标是构建一个面向科研场景的数据采集与分析平台，通过持续爬取 GitHub 项目数据，提供多维分析、自然语言检索和趋势预测能力。

### 1.3 项目目标

- 服务高校科研工作者，辅助研判软件产业技术趋势。
- 持续采集 GitHub 开源项目数据。
- 构建高效的数据存储与管理体系。
- 提供科研级多维分析能力。
- 支持自然语言检索与语义匹配。
- 支持编程语言与技术栈发展趋势预测。
- 提供可视化分析界面与导出能力。

### 1.4 项目特点

- 数据持续更新：基于定时调度实现增量采集。
- 数据规模大：覆盖多语言、多组织、多领域开源项目。
- 技术栈清晰：Python + Java + MongoDB + 前后端分离。
- 分析维度丰富：项目、组织、语言、品类、趋势等。
- 强检索能力：支持自然语言检索、关键词检索、相似项目检索。
- 科研导向：强调指标可解释性、结果可复现性和核验机制。

## 3. 系统架构设计

### 3.1 总体架构

```text
数据源（GitHub API / 页面）
  -> 采集层（Scrapy/Playwright）
  -> 清洗与标准化
  -> MongoDB 存储层
  -> 分析层（统计分析 + 趋势预测）
  -> 检索层（关键词 + 向量检索 + LLM 重排）
  -> API 服务层（Spring Boot）
  -> 前端展示层（Vue + ECharts）
```

### 3.2 核心业务流程

1. 定时任务触发增量采集。
2. 按统一 Schema 清洗并写入 MongoDB。
3. 触发离线分析任务更新指标与趋势结果。
4. 用户通过前端进行筛选、分析和自然语言检索。
5. API 返回结果并记录检索日志与反馈用于后续优化。

### 4.3 索引设计

#### 4.3.1 Project 索引

- 唯一索引：{ github_id: 1 }
- 排序与筛选：{ language: 1, stars_count: -1 }
- 活跃度查询：{ updated_at: -1 }
- 组合查询：{ category: 1, language: 1, long_term_value: -1 }
- 文本检索：{ name: "text", description: "text", topics: "text" }
- 向量检索：embedding 字段（MongoDB Atlas Vector Search）

#### 4.3.2 其他索引

- Commit：{ project_id: 1, commit_date: -1 }, { sha: 1 }（唯一）
- Contributor：{ project_id: 1, contributions: -1 }
- SearchRecord：{ created_at: -1 }，TTL 索引（如保留 180 天）

## 5. 功能模块设计

### 5.1 数据采集模块

- GitHub API 采集：项目、Owner、提交、贡献者基础数据。
- 页面采集：补充 API 不完整字段（按需启用）。
- 增量采集：按 updated_at + 游标机制拉取。
- 限流策略：令牌桶 + 多 Token 池轮转 + 重试退避。
- 调度机制：Quartz 定时任务，支持手动重跑与失败补偿。

### 5.2 数据清洗与加工模块

- 字段标准化：统一命名、时间格式、枚举值。
- 去重策略：按 github_id、sha、full_name 去重。
- 异常处理：空值修复、脏数据隔离、错误日志记录。
- 衍生指标计算：项目活跃度、社区参与度、质量分。

### 5.3 自然语言检索模块（核心）

#### 5.3.1 检索链路

1. 查询解析：识别意图与实体（语言、时间、组织、品类）。
2. 结构化召回：先走关键词/过滤条件召回。
3. 向量召回：对 query 与项目文本向量做近邻检索。
4. 融合排序：BM25 分 + 向量相似度 + 质量分加权。
5. LLM 重排：对 TopK 结果做语义重排与解释生成。
6. 结果校验：规则校验（过滤条件一致性、时间范围一致性）。

#### 5.3.2 排序公式（示例）

$$
Score = 0.35 \cdot BM25 + 0.35 \cdot CosSim + 0.20 \cdot Quality + 0.10 \cdot Freshness
$$

#### 5.3.3 质量评估指标

- 准确率（Precision@K）
- 召回率（Recall@K）
- NDCG@K
- 用户反馈点击率与停留时长

### 5.4 科研分析模块

#### 5.4.1 编程语言趋势分析

- 统计各语言在时间窗口内的项目数、活跃项目数、星标增长率。
- 采用时间序列方法（如 Prophet/ARIMA）做短期预测。
- 输出趋势斜率、波动率、置信区间。

#### 5.4.2 组织活跃度分析

- 活跃度评分：提交频率 + 更新频率 + 贡献者活跃度。
- 技术深度评分：技术栈广度与核心技术集中度。
- 支持组织对比与排行榜。

#### 5.4.3 软件品类分析

- 品类分类：Web、移动、桌面、工具、基础设施等。
- 占比与变化趋势分析。
- 与语言、组织维度做交叉分析。

#### 5.4.4 项目价值评估

长期价值评分示例：

$$
LTV = 0.30 \cdot Activity + 0.25 \cdot Community + 0.20 \cdot Maintainability + 0.15 \cdot Popularity + 0.10 \cdot Security
$$

## 6. API 设计

### 6.1 项目 API

- GET /api/projects：分页查询，支持语言、星标、活跃度筛选。
- GET /api/projects/{id}：项目详情。
- GET /api/projects/{id}/contributors：项目贡献者列表。
- GET /api/projects/{id}/commits：项目提交历史。

### 6.2 分析 API

- GET /api/analytics/language-trends?from=&to=
- GET /api/analytics/org-activity?from=&to=
- GET /api/analytics/category-distribution?from=&to=
- GET /api/analytics/project-value/{id}

### 6.3 检索 API

- POST /api/search/nl：自然语言检索。
- POST /api/search/keyword：关键词检索。
- GET /api/search/history：检索历史。

## 8. 性能设计

### 8.1 采集性能

- 异步抓取 + 并发控制。
- 增量采集减少冗余请求。
- 失败任务重试与断点续跑。

### 8.2 数据库性能

- 索引命中优先，避免全表扫描。
- 批量写入（bulkWrite）。
- 聚合查询结果缓存。

### 8.3 API 与前端性能

- API 分页与字段裁剪。
- Redis 缓存热点接口（可选）。
- 前端路由懒加载、图表按需加载、虚拟滚动。

## 9. 安全设计

### 9.1 数据与访问安全

- JWT 鉴权 + RBAC 权限控制。
- 敏感配置通过环境变量与密钥管理注入。
- 关键数据传输使用 HTTPS。

### 9.2 API 安全

- 输入参数校验与统一异常处理。
- 接口限流与防刷。
- CORS 白名单控制。

### 9.3 供应链安全

- 定期依赖漏洞扫描（SCA）。
- 镜像漏洞扫描与最小化镜像。

## 13. 风险与应对

- GitHub API 限流风险：多 Token 池、错峰调度、缓存与补采。
- 大模型成本风险：分层调用策略，优先结构化检索，LLM 仅用于重排与解释。
- 数据质量风险：引入抽样审核、规则校验、回溯修复机制。
- 预测偏差风险：持续评估模型误差并定期重训。
